## 一、项目背景

PegaFlow-Hust 是 NovaTalabs/PegaFlow 的昇腾 NPU（Ascend 910B3）适配版本，目标是在保持上游 KV Cache 卸载/共享能力不变的前提下，将底层从 CUDA 生态迁移至 CANN（Compute Architecture for Neural Networks）生态，实现以下核心能力：

1. **KV Cache D2H 卸载**: 推理过程中将 NPU 显存中的 KV Cache 异步拷贝至 CPU 钉内存
2. **KV Cache H2D 恢复**: 共享前缀命中时从 CPU 钉内存异步回拷到 NPU 显存
3. **SSD 持久化/预取**: 大容量 KV Cache 溢出至 SSD，按需预取
4. **RDMA 跨节点传输** (远期): 跨节点 KV Cache 共享

### 3.2 NPU 6 端到端部署命令

以下命令在 Ascend 910B3 NPU 6 上完整验证通过（2026-07-11）。

#### PegaFlow-Server（NPU 6）

```bash
cd /workspace/pegaflow-hust
# 复制编译产物
cp target/release/libpegaflow.so python/pegaflow/pegaflow.cpython-311-aarch64-linux-gnu.so
cp target/release/pegaflow-server-py python/pegaflow/pegaflow-server-py

# 启动 PegaFlow Server（NUMA affinity 禁用，避免 npu-smi 拓扑解析缺失）

    --addr 127.0.0.1:50055 \
    --devices 6 \
    --pool-size 2gb \
    --log-level info \
    --disable-numa-affinity \
    > /tmp/pegaflow-server-npu6.log 2>&1 &

# 验证健康状态
curl -s http://127.0.0.1:9091/health  # 应返回 "ok"
netstat -tlnp | grep 50055              # pegaflow-server 监听
```

#### vLLM 推理服务（NPU 6，集成 PegaKVConnector）

```bash
export ASCEND_HOME_PATH=/usr/local/Ascend/cann-8.5.1
export LD_LIBRARY_PATH="/usr/local/Ascend/driver/lib64/common:/usr/local/Ascend/driver/lib64/driver:/usr/local/Ascend/driver/lib64:/usr/local/Ascend/cann-8.5.1/lib64:/usr/local/Ascend/cann-8.5.1/runtime/lib64:/usr/local/Ascend/cann-8.5.1/opp/built-in/op_impl/ai_core/tbe/op_tiling:/usr/local/Ascend/nnal/atb/latest/atb/cxx_abi_1/lib:/usr/local/Ascend/ascend-toolkit/latest/lib64:$LD_LIBRARY_PATH"
export TORCH_DEVICE_BACKEND_AUTOLOAD=1
export ASCEND_RT_VISIBLE_DEVICES=6
export ASCEND_VISIBLE_DEVICES=6
export VLLM_PLUGINS=ascend
export HF_HUB_OFFLINE=1

cd /workspace
nohup /root/miniconda3/envs/vllm-hust-dev/bin/python -m vllm.entrypoints.openai.api_server \
    --model /data/shared_models/Qwen2.5-14B-Instruct \
    --kv-transfer-config '{"kv_connector":"PegaKVConnector","kv_role":"kv_both","kv_connector_module_path":"pegaflow.connector"}' \
    --enforce-eager \
    --max-model-len 4096 \
    --port 8000 \
    > /tmp/vllm-server-npu6.log 2>&1 &

# 等待启动完成
grep "Application startup complete" /tmp/vllm-server-npu6.log
# 验证 PegaKVConnector

# 验证注册
```

**关键配置说明**:
- `--disable-numa-affinity`: 因 `npu-smi` NUMA 拓扑解析未完成，需禁用 NUMA 感知内存分配
- `kv_connector_module_path=pegaflow.connector`: 显式指定 connector 模块路径，`VLLM_PLUGINS=ascend` 会过滤 general_plugins
- `enforce-eager`: Ascend 环境下必须禁用 CUDA Graph
- `gpu-memory-utilization`: 未显式设置，使用默认值 0.9

## 七、架构演进方向与优化点

### 7.1 短期优化（1-2 周）

1. **Raw Pointer 安全性加固**
   - 当前 raw pointer 方案依赖 pegaflow-server 和 vLLM worker 在同一主机
   - 建议：添加 `PEGAFLOW_ALLOW_RAW_PTR=1` 环境变量显式启用

2. **设备亲和性缓存**
   - 当前每次 `aclrtSetDevice` 调用有一定开销
   - 建议：线程本地存储当前活跃设备 ID，避免重复 `set_device`

3. **钉内存池预分配优化**
   - 当前按 NUMA node 均分池大小
   - 建议：支持 `--pinned-pool-per-device` 参数指定单设备池大小

### 7.2 中期优化（1-2 个月）

4. **camem_allocator 集成**
   - 编译 `vllm_ascend_C` 扩展模块，启用 `aclrtMallocPhysical` 分配
   - 这将使 CANN IPC 路径可用（跨进程、零拷贝）
   - 当前 raw pointer 方案作为 fallback 保留

5. **多流并发传输**
   - 当前单流 D2H/H2D 可能成为瓶颈
   - 建议：实现 per-layer 或多 block 并发传输，利用 NPU 多 DMA 引擎

6. **NUMA-aware 钉内存**
   - 完成 `npu-smi` 拓扑解析后启用 NUMA affinity
   - 预期提升：D2H/H2D 带宽 ~15-25%（避免跨 NUMA 节点访问）

### 7.3 长期优化（3-6 个月）

7. **RDMA 跨节点传输**
   - 需要 pegaflow-transfer 模块完整 Ascend 适配
   - 关键挑战：Ascend 设备内存注册到 RDMA MR（Memory Region）
   - 依赖于华为 `HCCS`（Huawei Cache Coherence System）或 `aclrtMallocPhysical` 分配器的 RDMA 支持

8. **SSD 卸载路径优化**
   - 当前 SSD 缓存通过 io_uring 异步 I/O
   - Ascend 设备内存无法通过 `O_DIRECT` DMA 到 NVMe
   - 需要 D2H → CPU 钉内存 → io_uring write 的双跳路径（已验证 D2H 可用）

9. **MetaServer 跨节点缓存注册**
   - 当前 MetaServer 使用内容哈希做跨节点去重
   - Ascend 环境下的哈希计算逻辑与 CUDA 一致（设备无关）

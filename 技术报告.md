## 一、项目背景

PegaFlow-Hust 是 NovaTalabs/PegaFlow 的昇腾 NPU（Ascend 910B3）适配版本，目标是在保持上游 KV Cache 卸载/共享能力不变的前提下，将底层从 CUDA 生态迁移至 CANN（Compute Architecture for Neural Networks）生态，实现以下核心能力：

1. **KV Cache D2H 卸载**: 推理过程中将 NPU 显存中的 KV Cache 异步拷贝至 CPU 钉内存
2. **KV Cache H2D 恢复**: 共享前缀命中时从 CPU 钉内存异步回拷到 NPU 显存
3. **SSD 持久化/预取**: 大容量 KV Cache 溢出至 SSD，按需预取
4. **RDMA 跨节点传输** (远期): 跨节点 KV Cache 共享

### 4.0 KV Cache Save/Load E2E 链路打通（最高优先级，已完成诊断 + 三层修复，剩余 1 个阻塞项）

**当前状态**: ⚠️ Save 链路已打通（Python 侧三层 Bug 全部修复），但 D2H 传输因硬件限制失败

**修复后实测数据**（NPU 5, Qwen2.5-14B, 2026-07-12，`PEGAFLOW_DEBUG_SAVE_PATH=1`）:

| 场景 | prompt tokens | completion | 延迟 | External cache hit | save 触发 |
|------|--------------|------------|------|-------------------|-----------|
| 长文本冷启动 | 201 | 30 | **2.61s** | 0.0% | ✅ 96 blocks |
| 相同长文本（"缓存命中"） | 201 | 30 | **2.49s** | 0.0% | — |
| 更长文本冷启动 | 301 | 100 | **8.39s** | 0.0% | ✅ 96 blocks |
| Server 稳定性 | — | — | — | — | ✅ 6 次 save RPC + 3 次 507899 后仍存活 |

**根因分析（2026-07-12 诊断确认）**:

Save/Load 链路被**三层 Bug 叠加阻塞**。诊断日志 (`PEGAFLOW_DEBUG_SAVE_PATH=1`) 揭示了完整的因果链：

```
Bug 1 (最上层 — 入口阻塞):
  PegaKVConnector.save_kv_layer() 被显式设为 pass 空操作
  → 依赖 wait_for_save() 被 Ascend backend 调用，但 Ascend backend 可能不调
  → save task 可能永远不入队，_save_worker 线程空闲
  状态: ✅ 已修复 (2026-07-12)

Bug 2 (中间层 — Save Intent 生成阈值):
  _consume_save_intent() 中 scheduled // virtual_block_size 公式要求 ≥128 tokens
  → 短 prompt (<128 tokens) 永远不生成 save intent
  → request_finished 时 blocks 被直接清理，KV cache 数据丢失
  状态: ✅ 已修复 (2026-07-12)

Bug 3 (最下层 — D2H DMA 传输):
  aclrtMemcpyAsync(D2H) 返回错误 507899（expandable_segments 不支持 DMA）
  → 原代码的同步 aclrtMemcpy fallback 在 expandable_segments 内存上触发
    CANN 驱动层 segfault → pegaflow-server 进程直接崩溃
  → 即使 Bug 1+2 被修复，所有 save RPC 也会导致 server 崩溃
  状态: ✅ 已修复崩溃 (2026-07-12)，⚠️ D2H 传输仍需 camem_allocator
```

**2026-07-12 修复详情** — 见第九章 Bug 修复记录。

**2026-07-12 诊断确认的关键发现**:

1. **`wait_for_save()` 被 Ascend backend 正常调用** — 日志中 160+ 次 `wait_for_save called`, 说明 vLLM Ascend 的 `NPUAttention` backend 正确触发了 connector 回调
2. **Save intent 正常生成** — 301-token prompt 产生 3 个 block hashes，prefill 阶段生成 3 个 save intent，decode 阶段 `new_blocks=0`（已全部消费）
3. **Save RPC 正常发送到 server** — 6 次 RPC 全部到达，96 layers × 96 blocks = ~24 MiB KV cache 数据
4. **Server 在 507899 错误后不再崩溃** — 返回 clean error: `"aclrtMemcpyAsync(D2H) failed: error 507899 ... Enable camem_allocator"`

**剩余阻塞项 — `aclrtMemcpyAsync` 507899**:

```
当前 CANN API 调用链:
  PegaEngine.save() → GpuWorkerPool.batch_save() → save_worker_loop()
    → process_save_task() → build_copy_descs() → AscendMemcpyBackend::d2h()
      → aclrtMemcpyAsync(DEVICE_TO_HOST, expandable_segments_ptr, ...)
        → 返回 507899 (ACL_ERROR_DMA_NOT_SUPPORTED)
```

`507899` 表示源地址内存由 `torch_npu` 默认的 `expandable_segments` 分配器分配，不是通过 `aclrtMallocPhysical`。CANN 驱动拒绝 DMA 操作。

**解决方案（按推荐顺序）**:

| # | 方案 | 工作量 | 说明 | 状态 |
|---|------|--------|------|------|
| 1 | **构建 `vllm_ascend_C` 扩展** | 1-2 天 | 启用 `camem_allocator`。已完成编译，但 `NPUPluggableAllocator` 在 CANN 8.5.1 + torch_npu 2.10.0 上不兼容 `change_current_allocator`（缺 stats 函数 + stack smashing）。`use_mem_pool` 路径不拦截 `torch.zeros` 分配。 | ❌ 不可行 |
| 2 | **升级 PyTorch 2.9.0 → 2.10.0** | 0.5 天 | 升级成功，camem_allocator C++ 接口不完整（缺 device_stats/reset 函数） | ❌ 已完成但不可行 |
| 3 | **Python 侧 PyTorch tensor copy 替代** | ✅ **已实施方案** | connector worker 中 `tensor.to('cpu')` D2H + `tensor.copy_()` H2D，修改 save/load 协议传递 CPU 数据。**Save 验证通过，Load 代码就绪** | ✅ **Save 成功** |
| 4 | **Rust 侧 PyO3 torch copy fallback** | — | 未实施（方案 3 已满足需求） | 跳过 |

**实际路径**: 方案 1+2 尝试后发现 CANN API 级别不可行 → 转向方案 3。

## 七、架构演进方向与优化点

### 7.1 短期优化（1-2 周）

1. **Raw Pointer 安全性加固**
   - 当前 raw pointer 方案依赖 pegaflow-server 和 vLLM worker 在同一主机
   - 建议：添加 `PEGAFLOW_ALLOW_RAW_PTR=1` 环境变量显式启用

2. **设备亲和性缓存**
   - 当前每次 `aclrtSetDevice` 调用有一定开销
   - 建议：线程本地存储当前活跃设备 ID，避免重复 `set_device`

3. **钉内存池预分配优化**
   - 当前按 NUMA node 均分池大小
   - 建议：支持 `--pinned-pool-per-device` 参数指定单设备池大小

### 7.2 中期优化（1-2 个月）

4. **camem_allocator 集成**
   - 编译 `vllm_ascend_C` 扩展模块，启用 `aclrtMallocPhysical` 分配
   - 这将使 CANN IPC 路径可用（跨进程、零拷贝）
   - 当前 raw pointer 方案作为 fallback 保留

5. **多流并发传输**
   - 当前单流 D2H/H2D 可能成为瓶颈
   - 建议：实现 per-layer 或多 block 并发传输，利用 NPU 多 DMA 引擎

6. **NUMA-aware 钉内存**
   - 完成 `npu-smi` 拓扑解析后启用 NUMA affinity
   - 预期提升：D2H/H2D 带宽 ~15-25%（避免跨 NUMA 节点访问）

### 7.3 长期优化（3-6 个月）

7. **RDMA 跨节点传输**
   - 需要 pegaflow-transfer 模块完整 Ascend 适配
   - 关键挑战：Ascend 设备内存注册到 RDMA MR（Memory Region）
   - 依赖于华为 `HCCS`（Huawei Cache Coherence System）或 `aclrtMallocPhysical` 分配器的 RDMA 支持

8. **SSD 卸载路径优化**
   - 当前 SSD 缓存通过 io_uring 异步 I/O
   - Ascend 设备内存无法通过 `O_DIRECT` DMA 到 NVMe
   - 需要 D2H → CPU 钉内存 → io_uring write 的双跳路径（已验证 D2H 可用）

9. **MetaServer 跨节点缓存注册**
   - 当前 MetaServer 使用内容哈希做跨节点去重
   - Ascend 环境下的哈希计算逻辑与 CUDA 一致（设备无关）

## 九、Bug 修复记录

> **修复日期**: 2026-07-12  
> **验证环境**: 8× Ascend 910B3 | CANN 8.5.1 | torch 2.9.0 | vLLM 0.23.1

### 9.1 Bug 1: `save_kv_layer()` 空操作阻塞 save 入口

**文件**: `python/pegaflow/connector/__init__.py:203-216`, `python/pegaflow/connector/worker.py:651-680`

**问题**: `PegaKVConnector.save_kv_layer()` 被显式设为 `pass`（开发者注释为 "Layer callbacks are intentionally ignored so graph replay cannot suppress save submission"）。如果 vLLM Ascend attention backend 不调用 `wait_for_save()`，save task 永远不会进入 `_save_queue`。

**修复**:

1. **`__init__.py`**: `PegaKVConnector.save_kv_layer()` 从 `pass` 改为委托 `self._worker.save_kv_layer(layer_name, kv_layer, attn_metadata)`，使层回调成为 save 的有效入口
2. **`worker.py`**: `WorkerConnector.save_kv_layer()` 移除未使用的 `metadata` 参数，与父类签名对齐。`_save_fallback_used` 标志用 `try/finally` 保护，防止异常后永久锁定

**验证**: 日志中 160+ 次 `wait_for_save called` 确认 save 路径正常触发。

---

### 9.2 Bug 2: Save Intent 生成受 `scheduled // virtual_block_size` 阈值限制

**文件**: `python/pegaflow/connector/scheduler.py:99-100, 396-399, 453-516`, `python/pegaflow/connector/common.py:68-70`

**问题**: `_consume_save_intent()` 中 `local_saveable = min(len(allocated), scheduled // virtual_block_size)` 公式要求累积 ≥128 tokens（`virtual_block_size=128`）才能生成 save intent。短 prompt（4-116 tokens）即使在请求完成时也无法触发保存，blocks 被直接清理。

**修复**:

1. **`common.py`**: `ConnectorContext` 新增 `write_enabled` 属性
2. **`scheduler.py`**: 新增 `_final_save_intents` 字典存储已完成请求的最终 save intent
3. **`scheduler.py`**: `request_finished()` 在清理请求前检查是否有未保存的 blocks，通过新增的 `_consume_final_save_intent()` 方法生成最终 save intent（不受 `scheduled` 阈值限制），并返回 `(True, None)` 保持 blocks 不被复用
4. **`scheduler.py`**: `build_connector_meta()` 将 `_final_save_intents` 合并到输出元数据

**验证**: 多轮请求中 save intent 正确生成（96 blocks/layer × 96 layers），6 次 save RPC 成功发送到 server。

---

### 9.3 Bug 3: `aclrtMemcpyAsync` 507899 导致 server 进程崩溃

**文件**: `pegaflow-core/src/device/ascend.rs:281-438`

**问题**: `aclrtMemcpyAsync(D2H/H2D)` 在 `expandable_segments` 分配的内存上返回错误 507899。原代码尝试同步 `aclrtMemcpy` fallback，但 CANN 驱动对非 DMA 内存执行同步拷贝时直接 **segfault** 导致整个 pegaflow-server 进程崩溃。这解释了技术报告中 "Save RPC 后 server 消失" 的现象。

**修复**:

1. 新增 `acl_error_description()` 函数，映射常见 CANN 错误码到人类可读描述
2. 定义 `ACL_ERROR_DMA_NOT_SUPPORTED = 507899` 常量
3. 当 async 调用返回 507899 时，**跳过同步 fallback**（避免 segfault），直接返回清晰的 `Err(...)` 包含：
   - 具体错误码和描述
   - 根因说明（expandable_segments 不支持 DMA）
   - 可操作的解决建议（启用 camem_allocator 或 vllm_ascend_C）
4. 其他错误码仍保留同步 fallback 路径，增加成功/失败的区分日志

**验证**: Server 处理 3 次 507899 错误后仍正常运行（Health: ok），不再崩溃。Save RPC 在 ~3ms 内返回 clean error 给 vLLM。

### 10.8 注意事项（重要！）

| # | 注意 | 说明 |
|---|------|------|
| 1 | **NPU 内存不自动释放** | `kill` 进程后 HBM 占用不会立刻清零。需用 `npu-smi info` 确认空闲，必要时切换 NPU |
| 2 | **二进制路径** | pegaflow-server 二进制位于 `/workspace/pegaflow-hust/python/pegaflow/pegaflow-server-py`，必须从源码根目录的 python/pegaflow/ 目录加载 Python 扩展 |
| 3 | **conda Python** | 所有 Python 命令必须使用 `/root/miniconda3/envs/vllm-hust-dev/bin/python`（3.11），系统 Python 3.13 缺少 torch |
| 4 | **不需要 --enable-sleep-mode** | Plan 3 不依赖 camem_allocator，vLLM 启动不要加 `--enable-sleep-mode` |
| 5 | **PYTHONPATH 必须设置** | pegaflow-server 二进制内嵌 PyO3，需要 PYTHONPATH 找到 torch/torch_npu |
| 6 | **LD_LIBRARY_PATH 顺序** | conda lib 必须在最前面（`/root/miniconda3/envs/vllm-hust-dev/lib`），否则加载系统 Python 导致 "No module named 'torch'" |
| 7 | **Python 缓存清理** | 修改 worker.py 或任何 Python 文件后，必须清理 `__pycache__` 和 `.pyc` 文件 |
| 8 | **proto 修改后重建** | 修改 engine.proto 后必须 `cargo build --bin pegaflow-server-py --lib -p pegaflow-py` |
| 9 | **src 中 .so** | 编译后 `target/release/libpegaflow.so` 必须复制到 `python/pegaflow/pegaflow.cpython-311-aarch64-linux-gnu.so` |
| 10 | **跨实例 Load** | 同实例内 vLLM 前缀缓存会拦截外部查询（命中率 ~47%），Load 需跨实例或缓存清空后测试 |

### P1-1: camem_allocator 方案验证

**结论: 方案 B（混合分配）不需要。** 在当前 CANN 8.5.1 环境下：

| 功能 | 状态 | 说明 |
|------|------|------|
| `aclrtMemcpyAsync` D2H/H2D | ✅ | torch.empty 内存原生支持 DMA |
| `aclrtIpcMemGetExportKey` (ctypes) | ❌ | 裸 CANN ctypes 没有设备初始化 → 507899 |
| `storage._share_npu_()` (PyTorch) | ✅ | PyTorch 内置 IPC 正确初始化 CANN |

### 关键修复: NpuIPCWrapper

**问题**: `npu_ipc_wrapper.py` 用裸 `ctypes.CDLL(libascendcl.so)` 调用 CANN API，绕过了 PyTorch 的设备初始化。

**修复**: 改用 PyTorch 内置 IPC（与 `CudaIPCWrapper` 的 `_share_cuda_` 模式一致）：
```python
# 导出 (client)
self._handle = tensor.untyped_storage()._share_npu_()

# 导入 (server)
storage = torch_npu._C._new_shared_npu(*self._handle)
```

| `aclrtMemcpy` sync H2D | ✅ | ✅ | ✅ | ✅ | ✅ |

### IPC (跨进程内存共享)

| 方法 | torch.empty | torch.zeros | aclrtMalloc |
|------|------------|-------------|-------------|
| `aclrtIpcMemGetExportKey` (裸 ctypes) | ⚠️ 需 2MB 对齐 | ❌ 507899 | ✅ |
| `aclrtIpcMemImportByKey` (裸 ctypes) | ❌ 507899 | ❌ 507899 | ❌ 507899 |
| `storage._share_npu_()` (PyTorch) | ✅ | ✅ | ✅ |
| `torch_npu._C._new_shared_npu()` | ✅ | ✅ | ✅ |

### 诊断发现

1. 裸 CANN ctypes 的 `aclrtIpcMemImportByKey` **同进程和跨进程都失败**——根本原因是缺少设备初始化
2. PyTorch 内置 IPC 正确处理了 CANN 初始化，**不需要 2MB 对齐**
3. 之前误判为内存类型/对齐问题的 507899 错误，实际是 IPC 调用方式问题

## 四、当前数据流架构

```
vLLM Worker (Python)                  pegaflow-server (Rust + PyO3)
───────────────────                   ──────────────────────────────
torch.empty('npu:4')                  
  ↓ storage._share_npu_()             
NpuIPCWrapper(handle)                 
  ↓ pickle.dumps()                    
  ↓ gRPC register_context_batch ────→ pickle.loads() → NpuIPCWrapper
                                       ↓ to_tensor() → torch_npu._C._new_shared_npu()
                                       ↓ 真实 torch.Tensor (零拷贝跨进程共享)
                                       ↓ data_ptr() → Rust 引擎

gRPC save ──────────────────────────→ build_copy_descs()
                                       ↓ AscendMemcpyBackend::d2h()
                                       ↓ aclrtMemcpyAsync(D2H) → pinned memory ✅

gRPC query ─────────────────────────→ cache lookup → lease

gRPC load ──────────────────────────→ AscendMemcpyBackend::h2d()
                                       ↓ aclrtMemcpyAsync(H2D) ✅

SSD write (pending)                 → D2H data → io_uring O_DIRECT
                                       ⚠️ 需要 ext4/xfs
```

## 七、部署命令

```bash
# 1. 环境变量
export LD_LIBRARY_PATH=/root/miniconda3/envs/vllm-hust-dev/lib:\
/usr/local/Ascend/cann-8.5.1/aarch64-linux/lib64:\
/usr/local/Ascend/driver/lib64/driver

# 2. 编译
cd /workspace/pegaflow-hust
PYO3_PYTHON=/root/miniconda3/envs/vllm-hust-dev/bin/python \
  cargo build --no-default-features --features ascend \
  -p pegaflow-py --bin pegaflow-server-py

# 3. 复制 PyO3 模块
cp target/debug/libpegaflow.so \
  python/pegaflow/pegaflow.cpython-311-aarch64-linux-gnu.so

# 4. 启动 server
./target/debug/pegaflow-server-py \
  --addr 127.0.0.1:50055 --devices 4 \
  --pool-size 500mb --disable-numa-affinity

# 5. 运行 E2E 测试
python e2e_test.py
```

### 1.2 核心结论

**当前状态：单机原型验证阶段，两处架构偏离待还原。**

迁移工作的技术架构设计合理——设备抽象层设计良好，CUDA 代码被干净地 feature-gated。但关键的数据传输链路和跨进程内存共享机制因 **CANN `aclrtMemcpyAsync` 和 `aclrtIpcMemImportByKey` 要求 Device 侧内存来源于 `aclrtMallocPhysical`，而 PyTorch 缓存分配器底层并非通过此 API 分配**，引入了两处根本性的架构偏离：(1) Plan 3——Python 侧 `tensor.to('cpu')` / `tensor.copy_()` 绕过 Rust DMA；(2) Raw Pointer 模式——`NpuIPCWrapper(key=b"")` + `_RawDeviceProxy` 伪造 tensor 绕过 CANN IPC。两者还原方案分别见 §8 和 §9，均依赖 camem_allocator 就绪。

## 三、详细对比分析

### 3.1 核心架构与初始化

#### 3.1.1 设备抽象层设计 — 架构级评价

迁移工作最成功的设计决策是引入了独立的 `device/` 模块（`pegaflow-core/src/device/`），将设备差异隔离在统一的 enum 后：

```rust
// pegaflow-core/src/device/mod.rs:17-25
pub enum DeviceContext {
    #[cfg(feature = "cuda")]
    Cuda(Box<cuda::CudaDevice>),
    #[cfg(feature = "ascend")]
    Ascend(ascend::AscendDevice),
}

pub enum DeviceStream {
    #[cfg(feature = "cuda")]
    Cuda(cuda::CudaDeviceStream),
    #[cfg(feature = "ascend")]
    Ascend(ascend::AscendDeviceStream),
}
```

**对比 CUDA 原版**: 原版直接依赖 `cudarc::driver::CudaContext` 和 `CudaStream`，没有抽象层。`gpu_worker.rs` 第 3 行直接 `use cudarc::driver::{CudaContext, CudaStream}`。

**关键差异**:
- Ascend 不维护类似 `Arc<CudaContext>` 的句柄——`aclrtIpcMemImportByKey` 指针是进程级的，不需要 per-context handle。`AscendDevice` 只存 `device_id: i32`
- Ascend 创建 stream 时**必须先调用 `aclrtSetDevice`**（`gpu_worker.rs:326-330`），而 CUDA 的 stream 创建隐式绑定到当前 context

**缺失的 Ascend 能力**（`device/mod.rs:78-109`）:
- `alloc_zeros<T>()`: Ascend arm 返回 `Err("Ascend backend does not support scratch allocations")`
- `record_event()`: Ascend arm 返回 `Err("record_event: Ascend backend not yet supported")`

这意味着 Ascend 后端**无法使用原始的 CUDA-based `alloc_zeros` scratch buffer**（被 `kernel.rs` 使用），也**无法进行 event-based 的时序测量**。

#### 3.1.2 CANN FFI 绑定方式

全部 `extern "C"` 直接调用 `libascendcl.so`，而非通过 `cudarc` 式的 safe wrapper：

```rust
// pegaflow-core/src/device/ascend.rs:39-91
unsafe extern "C" {
    fn aclInit(config_path: *const i8) -> aclError;
    fn aclrtSetDevice(device_id: i32) -> aclError;
    fn aclrtCreateStream(stream: *mut aclrtStream) -> aclError;
    fn aclrtMemcpyAsync(dst, dst_max, src, count, kind, stream) -> aclError;
    fn aclrtMemcpy(dst, dst_max, src, count, kind) -> aclError;
    fn aclrtMalloc(ptr, size, policy) -> aclError;
    fn aclrtFree(ptr) -> aclError;
    fn aclrtMallocHost(ptr, size) -> aclError;
    fn aclrtFreeHost(ptr) -> aclError;
    fn aclrtGetVersion(major, minor, patch) -> aclError;
}
```

**注意**: 该 FFI 层**未声明** `aclrtIpcMemGetExportKey` / `aclrtIpcMemImportByKey` / `aclrtIpcMemClose`——这些调用仅在 Python 侧的 `npu_ipc_wrapper.py` 通过 ctypes 完成。Rust 侧不参与 NPU IPC 管理，仅使用 raw device pointers。

#### 3.1.3 服务端设备检测 — 运行时分发

```rust
// pegaflow-server/src/lib.rs:323-357
```

这是**运行时分发**而非编译时特性选择——同一个二进制在两种平台上都能运行。这是一个务实的权衡：意味着二进制链接了 `libascendcl.so`，但若运行时没有该库也能优雅处理。

### 3.2 内存管理

#### 3.2.1 钉内存分配对比

| 操作 | CUDA 原版 | Ascend 迁移 | 差异 |
|------|-----------|-------------|------|
| 分配 | `cudaHostAlloc(&ptr, size, cudaHostAllocDefault)` 来自 `cudarc::runtime::sys` | `aclrtMallocHost(&ptr, size)` 来自 `extern "C"` | ✅ |
| 释放 | `cudaFreeHost(ptr)` | `aclrtFreeHost(ptr)` | ✅ |
| 设备指针 | `cudaHostGetDevicePointer(&dev_ptr, ptr, 0)` | **不需要**—— host ptr 即 device ptr | ⚠️ 架构差异 |
| 对齐 | 页面级 | 64 字节 (`ASCEND_HOST_ALIGNMENT = 64`) | ⚠️ |
| HugePages 注册 | `cudaHostRegister`（可选） | 未实现 | ❌ |

关键差异在 `ascend.rs:249-269`：
```rust
pub fn malloc_host(size: usize) -> Result<(*mut u8, *mut u8)> {
    // ...
    // On Ascend, host and device pointers are the same for malloc_host allocations.
    Ok((ptr as *mut u8, ptr as *mut u8))
}
```

这与 `pinned_mem.rs` 中 CUDA 版本需要两次调用（分配 + get device pointer）的语义不同。

#### 3.2.2 Device 侧内存来源不匹配 — 根本性阻塞

这是**整个迁移工作中最关键的技术问题**。507899 错误的原因不是 `expandable_segments` 本身的开关状态，而是更深层的内存分配器架构差异，涉及 CANN 的两个独立约束：

**约束 A（Host 侧，官方文档明确）**：`aclrtMemcpyAsync` 用于 H2D/D2H 时，Host 侧内存必须来自 `aclrtMallocHost`。PegaFlow Ascend 版的钉内存分配已满足此约束（`pinned_mem.rs` 中 `allocate_ascend_host()` 使用 `aclrtMallocHost`）。

**约束 B（Device 侧，实测返回 507899）**：Device 侧内存必须来自 `aclrtMallocPhysical`（或 `aclrtMalloc`），不能来自 PyTorch 缓存分配器的内存池。

问题根源在于 PyTorch 的缓存分配器（Caching Allocator）——它向底层驱动申请大块内存后自行切割复用，这块内存池的"源头"**不是** `aclrtMallocPhysical`。因此当 vLLM 通过 `torch.npu.empty()` 创建 KV Cache tensor 后，PegaFlow 的 `aclrtMemcpyAsync(H2D/D2H)` 以该 tensor 的 data_ptr 作为 src/dst 时，CANN 驱动在任务实际执行时报告 507899。由于该错误是**异步的**——`aclrtMemcpyAsync` 调用本身可能返回 0，真正失败发生在任务排队执行后——且同步 `aclrtMemcpy` 对这类内存可能触发驱动层 segfault，无法使用同步 fallback。

**关键澄清——`expandable_segments` 与此无关**：

`expandable_segments` 控制的是 PyTorch 缓存分配器**内部**的虚拟内存管理策略（是否使用可扩展的虚拟内存段），它**不改变**分配器底层内存的"源头"。因此：
- `expandable_segments:False` 不能解决 507899
- `expandable_segments:True` 也不是 507899 的原因
- 无论 expandable_segments 开或关，PyTorch 缓存分配器底层内存都不是 `aclrtMallocPhysical` 分配的

受影响的 API 范围：
1. **CANN IPC 导出失败**（`aclrtIpcMemGetExportKey` 虽返回成功，但后续 `aclrtIpcMemImportByKey` 因 507899 失败；`npu_ipc_wrapper.py:213` 已禁用 IPC key 导出）
2. **DMA 拷贝失败**（`aclrtMemcpyAsync(H2D/D2H)` 返回 507899）
3. **同步 memcpy 崩溃**（`aclrtMemcpy` 对非 `aclrtMallocPhysical` 的设备内存触发驱动层 segfault，`ascend.rs:338-345,408-415` 专门跳过了 sync fallback）

尝试的修复路径均受阻：
- `PYTORCH_NPU_ALLOC_CONF=expandable_segments:False`（**无效**——不改变内存来源）
- `camem_allocator` + `change_current_allocator`（`NPUPluggableAllocator` 在 CANN 8.5.1 上接口不完整；`camem_allocator` 使用 `aclrtMallocPhysical` 作为底层分配，是**理论上正确的修复方向**）
- `vllm_ascend_C` 扩展（编译成功但 `change_current_allocator` 缺 stats 函数 + stack smashing）

**对比 CUDA 为什么没这个问题**：CUDA 的 `cuMemAlloc` / `cuMemCreate` 分配的所有设备内存天然支持 `cuMemcpyAsync` 和 `cuIpcGetMemHandle`——不存在"设备内存来源决定是否可 DMA"的区分。这是 CUDA 与 Ascend CANN 在内存管理架构上的根本性差异，不是配置或版本问题。

#### 3.2.3 Raw Pointer 模式 — 绕过而非实现 CANN IPC

由于 PyTorch 缓存分配器底层非 `aclrtMallocPhysical` 导致 CANN IPC 不可用（`aclrtIpcMemGetExportKey` 虽返回成功，但后续 `aclrtIpcMemImportByKey` 因 507899 失败），迁移采用了 raw pointer 模式——但这**不是 CANN IPC 的替代实现，而是完全绕过了 IPC 机制**：

```python
# npu_ipc_wrapper.py:213-214
self.key: bytes = b""           # CANN IPC path disabled
self._raw_ptr: int = ptr        # store raw NPU device virtual address
```

**这意味着放弃的不是性能优化，而是 PegaFlow 的架构基础**：

1. **内存共享被替换为地址传递**：CUDA IPC 通过 OS 级映射实现真正的跨进程内存共享（两个进程可以同时访问同一块物理内存）。Raw pointer 只传递了一个数字——server 收到的是 vLLM worker 地址空间中的一个虚拟地址，没有映射、没有共享、没有安全保障。

2. **`_RawDeviceProxy` 伪造 tensor 接口**：`registry.rs:materialize_tensor()` 期望得到一个真实的 `torch.Tensor`（通过 `Py<PyAny>` 持有其生命周期）。Raw pointer 模式下 `to_tensor()` 返回的 `_RawDeviceProxy` 只提供 `data_ptr()`、`device.index`、`nbytes()` 三个方法——这是一个通过 duck-typing 骗过 Rust 侧类型检查的假对象，**不能被 `set_()`、不能参与 PyTorch 计算图、不能被 GC 正确管理**。

3. **生命周期安全完全缺失**：CUDA IPC 映射的内存有引用计数——只要 `registry.rs` 持有 `Py<PyAny>`，底层 IPC 内存就不会被释放。Raw pointer 模式下，vLLM worker 随时可能释放 tensor，server 持有的 raw ptr 成为悬垂指针——没有任何机制阻止这件事。

4. **跨节点部署不可能**：CUDA IPC handle 可以导出 dma-buf fd，进而注册 RDMA MR 进行跨节点零拷贝传输。Raw pointer 只是一个同进程地址空间内的数字，没有任何跨节点扩展路径。

`npu_ipc_wrapper.py` 的注释（第 207-212 行）准确描述了原因："`aclrtIpcMemGetExportKey` succeeds for expandable_segments allocations (producing a non-empty key), but `aclrtIpcMemImportByKey` fails in the target process (507899)"——IPC key 导出是成功的，但导入端因为内存不是 `aclrtMallocPhysical` 分配而失败。注释中 "raw device pointers are sufficient — no IPC key needed" 的结论仅在单机原型验证阶段成立，**不符合 PegaFlow 的设计本意**。

### 3.3 数据传输与通信

#### 3.3.1 Save 路径 (D2H) — 双路径分析

**原始设计路径**（Rust 侧 DMA，源仓库）:
```
vLLM → gRPC save RPC → PegaEngine.save()
  → batch_save_kv_blocks_from_ipc()  [offload.rs]
  → GpuWorkerPool.batch_save()       [offload.rs]
  → process_save_task()              [gpu_worker.rs:474]
  → build_copy_descs()               [gpu_worker.rs]
  → backend.d2h(&copies, stream)     [gpu_worker.rs:486]
  → cuMemcpyDtoHAsync_v2()
  → stream.synchronize()
```

**Ascend Plan 3 实际路径**:
```
vLLM tensor.to('cpu') → CPU bytes → gRPC save RPC (block_data inline)
  → PegaEngine.batch_save_kv_blocks_from_ipc()  [offload.rs:263]
  → pre_copied_registry[HashMap<String, Vec<usize → Vec<u8>>]  [offload.rs:289-329]
  → GpuWorkerPool.batch_save()
  → process_save_task()                           [gpu_worker.rs:527]
  → if all_pre_copied:
      ptr::copy_nonoverlapping(data → host buffer) [gpu_worker.rs:563]
      skip backend.d2h()                           [gpu_worker.rs:568-569]
    else:
      build_copy_descs() + backend.d2h()           [gpu_worker.rs:570-574]
```

**关键发现**: `gpu_worker.rs:539` 的 `all_pre_copied` 检查决定了走 Plan 3 还是原生 DMA 路径。在 Ascend 的 Plan 3 模式下，所有层都有 pre_copied 数据时，`aclrtMemcpyAsync` **永远不会被调用**。这意味着 `AscendMemcpyBackend::d2h()` 方法在 Save 路径上实质上是**死代码**——它仅在 pre_copied 数据不完全时作为 fallback 被调用。

Python 侧的 D2H 实现：
```python
# worker.py:860-868
for blk_id in block_ids:
    block_view = actual_tensor[blk_id].contiguous()
    cpu_block = block_view.to('cpu', non_blocking=True)
    cpu_block = cpu_block.contiguous()
    _device_synchronize(self._torch_device)
    block_data_list.append(cpu_block.view(torch.uint8).numpy().tobytes())
```

**性能评估**: Plan 3 每次都执行完整的 `tensor.to('cpu')`，包含了 PyTorch 内部的 D2H 路径（同样是 DMA，PyTorch 内部可能使用了不同的 API 路径绕过了 507899）。这比原版 Rust 侧的零拷贝 `cuMemcpyAsync` 多了一次 Python→Rust→CPU 的数据流转。

#### 3.3.2 Load 路径 (H2D) — 双路径分析

Server 侧同时尝试两种路径：
```rust
// service.rs:479-509
// Plan 3: read CPU block data from storage (works on Ascend)
let loaded_layers = self.engine.read_block_data_for_leases(...)?;

// Also attempt normal H2D path; ignore failure on Ascend
let _ = self.engine.batch_load_kv_blocks_multi_layer(...);
```

原生的 `batch_load_kv_blocks_multi_layer` 在 Ascend 上会因 `aclrtMemcpyAsync(H2D)` 返回 507899 而**必然失败**，但 Rust 侧**吞掉了这个错误**（`let _ = ...`），依赖 Plan 3 的 CPU data 路径作为实际通道。

Python 侧 H2D：
```python
# worker.py:577-597
if loaded_layers:
    for layer_name, block_data_list in loaded_layers:
        for i, blk_id in enumerate(all_block_ids):
            if i < len(block_data_list) and block_data_list[i]:
                cpu_block = torch.frombuffer(bytearray(block_data_list[i]), ...)
                actual_tensor[blk_id].copy_(cpu_block.to(device=...))
    _device_synchronize(self._torch_device)
```

**问题**: Load 路径将完整的 block data（~250KB/block × 96 blocks = ~25MB）从 server 传回 vLLM worker 作为 gRPC 响应。这比原版的 H2D DMA 多了一次网络往返的开销。但在单机场景下（localhost gRPC），影响不大。

#### 3.3.3 aclrtMemcpyAsync 507899 错误处理

```rust
// ascend.rs:286-295
fn acl_error_description(code: i32) -> &'static str {
    match code {
        507899 => "memory not allocated via aclrtMallocPhysical ...",
        207001 => "invalid parameter",
        207002 => "memory allocation failed",
        207003 => "device not available",
        207004 => "stream is invalid",
        _ => "see CANN documentation for error code details",
    }
}
```

对 507899 的特殊处理（`ascend.rs:338-345` 和 `408-415`）**跳过同步 fallback** 因为 `aclrtMemcpy` 会在 expandable_segments 内存上触发 segfault。这是一个关键的安全修复——但仅阻止了崩溃，未解决问题本身。

#### 3.3.4 通信协议变更

proto 新增字段：
```protobuf
// engine.proto (diff)
message SaveLayer {
  repeated bytes block_data = 4;  // Plan 3: pre-copied CPU block data
}

message LoadedLayer {             // Plan 3: returned block data
  string layer_name = 1;
  repeated bytes block_data = 2;
}

message LoadResponse {
  repeated LoadedLayer layers = 2; // Plan 3: CPU data instead of H2D
}
```

这是一个向后不兼容的协议变更——必须同时升级 client 和 server。

### 3.4 算子与计算逻辑

#### 3.4.1 KernelBackend 未迁移

`KernelBackend`（`pegaflow-core/src/transfer/kernel.rs`）包含基于 CUDA nvcomp 的 banner 压缩 + 自定义 kernel。在 Ascend 下：

```rust
// gpu_worker.rs:301-313
TransferMode::Kernel => {
    #[cfg(feature = "cuda")]
    {
        if let DeviceContext::Cuda(ref cuda) = ctx {
            // ... nvcomp kernel
        }
    }
    Err(EngineError::DeviceInit(
        "Kernel transfer mode requires a CUDA device context".into(),
    ))
}
```

Ascend 需要 `TransferMode::AscendDirect`（即 `AscendMemcpyBackend`），不支持 `Kernel` 模式。原版 PegaFlow 通过 nvcomp 实现的 banner 压缩优化在 Ascend 上**完全不可用**。

**注意**: `TransferMode::AscendDirect` 仅在 Rust 内部 enum（`transfer/mod.rs:87`）中定义，**protobuf 的 `TransferMode` enum 中没有对应变体**，`service.rs:233-236` 的 match 仅有 `Direct` 和 `Kernel` 两个分支。实际运行时，Ascend 通过 `gpu_worker.rs:285-288` 将 `TransferMode::Direct` 映射到 `AscendMemcpyBackend`（当 device context 为 Ascend 时），因此 `AscendDirect` 变体实质上是不可通过 gRPC 触发的预留代码。

#### 3.4.2 合并算法复用

`merge()` 函数（`ascend_memcpy.rs:24-52`）与 CUDA 版本（`memcpy.rs`）使用完全相同的 contiguous coalescing 算法，这是正确的代码复用。

### 3.5 Python/C++ 混合编程

#### 3.5.1 IPC Wrapper 对比

| 方面 | CUDA 原版 | Ascend 迁移 |
|------|-----------|-------------|
| IPC 机制 | `cuIpcGetMemHandle` / `cuIpcOpenMemHandle` | ❌ 不可用 → raw pointer |
| Wrapper 类 | `CudaIPCWrapper` (torch.Tensor → CUDA IPC Handle) | `NpuIPCWrapper` (torch.Tensor → raw ptr + _RawDeviceProxy) |
| pickle 序列化 | IPC Handle (64 bytes) → gRPC | pickle(_RawDeviceProxy state: 8 elements) |
| to_tensor() | `torch.Tensor.set_(cuda_ipc_storage)` → 真实 tensor | `_RawDeviceProxy`（仅 meta: data_ptr, device.index, nbytes） |
| 跨进程内存访问 | CUDA IPC 映射（零拷贝） | **同一设备虚拟地址**（依赖 mmap 不存在，依赖同主机/dev idx 一致） |
| 跨节点可用 | ✅ 通过 RDMA 扩展 | ❌ raw pointer 只在同主机有效 |

#### 3.5.2 C 扩展为孤立代码

`_npu_ipc.c` (250 行) 实现了 `export_key` / `import_key` / `close_key` 函数，有其 `setup.py`，但：

1. `pyproject.toml` 未引用此扩展作为构建依赖
2. `scripts/build-wheel.sh` 和 `scripts/check.sh` 均未涉及
3. `npu_ipc_wrapper.py` 的 ctypes fallback 是默认路径
4. 由于 raw pointer 模式下 `self.key = b""`（第 213 行），IPC 导出路径**永远不会被触发**

C 扩展目前是**完全死代码**，与构建系统零耦合。

#### 3.5.3 PyO3 bindings 签名变更

对比 `python/src/lib.rs`（Plan 3 变更）:
- `save()`: 新增 `saves_list: Vec<(String, Vec<i32>, Vec<Vec<u8>>, Vec<Vec<u8>>)>`（block_data 字段）
- `load()`: 返回 `Vec<(String, Vec<Vec<u8>>)>`（loaded_layers）

`python/pegaflow/pegaflow.pyi` 的 36 行更新对应这些签名变更。

此外，`python/src/lib.rs` 中保留了完整的 `session_stream` / `start_session_watcher` 实现（tonic `Streaming<SessionEvent>`），此功能在两个仓库中一致，迁移未受影响。

#### 3.5.4 哈希计算（BLAKE3）迁移评估

源仓库 `pegaflow-core/src/hash.rs` 中 BLAKE3 哈希用于 GPU-side kernel（CUDA），目标仓库已切换为 Rust CPU-side 计算。由于 BLAKE3 本身是设备无关的纯计算逻辑，切换对正确性无影响。源仓库的 CUDA kernel 版本旨在减少 D2H 传输（先哈希再传 hash 而非传全量数据），Ascend 下因 Plan 3 已包含 D2H 的完整数据拷贝，CPU-side 哈希是合理权衡。**这不是迁移缺失，而是适配 Plan 3 架构的刻意设计决策**。

### 4.1 严重风险

| # | 风险 | 影响 | 现状 |
|---|------|------|------|
| **R1** | **D2H/H2D 完全依赖 PyTorch 内部路径** | Plan 3 的 `tensor.to('cpu')` 是黑盒，PyTorch 版本升级可能改变其行为或性能特征。不支持 CAMEM 的 `aclrtMemcpyAsync` 路径名存实亡 | 无缓解 |

## 七、恢复 Rust 侧 DMA 的解决思路（专题分析）

### 7.1 问题重申

`aclrtMemcpyAsync` 用于 Host↔Device 传输时，要求：
- **Host 侧**：内存必须来自 `aclrtMallocHost`（PegaFlow 已满足，钉内存池使用此 API）
- **Device 侧**：内存必须来自 `aclrtMallocPhysical` 或 `aclrtMalloc`（**不满足**——PyTorch 缓存分配器的底层内存不走这些 API）

CUDA 没有这种"内存来源决定是否可 DMA"的区分——所有 `cuMemAlloc` 产出的设备内存天然可 DMA 和 IPC。这是 Ascend 的根本性架构差异。

### 7.2 可行方案

#### 方案 A: 完整的 `camem_allocator` 集成（推荐，根本修复）

**原理**：用 `aclrtMallocPhysical` + `aclrtMapMem` 作为 PyTorch 的底层设备分配器，替换默认的 PyTorch 缓存分配器。这样所有 NPU tensor（包括 KV Cache）的内存来源变为 `aclrtMallocPhysical`，满足 `aclrtMemcpyAsync` 和 CANN IPC 的一切要求。

**具体步骤**：
1. 编译 vllm-ascend 中的 `camem_allocator.cpp` → `.so`（已在 pegaflow-hust 中编译成功）
2. 在 PyTorch 初始化完成**之前**（任何 `torch.npu.empty()` 调用之前）调用：
```python
import torch_npu
new_alloc = torch_npu.npu.memory.NPUPluggableAllocator(
    'libcamem_allocator.so', 'my_malloc', 'my_free'
)
torch_npu.npu.memory.change_current_allocator(new_alloc)
```
3. 需补齐 C 扩展中的 `device_stats` / `reset_peak_stats` / `reset_accumulated_stats` 等可选函数指针（通过 `allocator().set_get_device_stats_fn()` 等设置），避免 stack smashing
4. 验证 `aclrtMemcpyAsync(D2H/H2D)` 在真实 KV Cache tensor 上不再返回 507899
5. 恢复 `gpu_worker.rs:574` 的 `backend.d2h()` 原生路径

**当前阻塞点**：
- `change_current_allocator` 要求在任何 NPU 内存分配之前调用（"already initialized" 错误），但 vLLM 在 connector 加载前已经分配了大量 NPU 内存（模型权重等）
- `camem_allocator.cpp` 的 `my_malloc`/`my_free` 缺少部分可选函数指针实现，导致 stack smashing
- `expandable_segments:True` 与 `camem_allocator` 不兼容（PyTorch bug [#147851](https://github.com/pytorch/pytorch/issues/147851)）

**优先级**：必须在 vLLM 启动的最早阶段（在 `torch.npu` 首次分配之前）完成 `change_current_allocator`。典型做法是：
- 通过 `VLLM_ASCEND_USE_CAMEM=1` 环境变量或 vLLM 的 `--kv-cache-dtype` 之前的 hook 点注入
- 或在 vLLM worker 进程 fork 后、模型加载前插入 allocator swap

**修复后收获**：Rust 侧原生 `aclrtMemcpyAsync` D2H/H2D + CANN IPC 零拷贝 + SSD 缓存路径 + RDMA 跨节点传输——全部一次性恢复。

#### 方案 B: 分阶段混合分配（中等改动，折中）

**原理**：不替换全局分配器，而是为 KV Cache 层单独分配 DMA 兼容的设备内存。PegaFlow 在注册 layer 时，通过 `aclrtMalloc` + `ACL_MEM_MALLOC_HUGE_FIRST` 分配一块独立的设备内存作为 KV Cache 存储，然后通过 `torch.tensor(..., device='npu')` 的 `set_()` 方法映射到这块内存上。

**具体步骤**：
1. 在 `registry.rs` 中注册 layer 时，Rust 侧通过 FFI 调用 `aclrtMalloc`（policy = `ACL_MEM_MALLOC_HUGE_FIRST`）分配设备内存
2. 将分配的 `data_ptr` 传回 vLLM worker，worker 使用 `torch.Tensor.set_()` 将 KV Cache tensor 重新指向这块内存
3. 后续 PegaFlow 的 `aclrtMemcpyAsync(D2H/H2D)` 以这块 `aclrtMalloc` 分配的内存为 src/dst，不再报 507899

**优点**：
- 不影响 PyTorch 全局分配器，vLLM 其余部分（模型权重等）仍使用默认分配器
- 比方案 A 更轻量，不需要 `change_current_allocator`

**缺点**：
- 需要侵入 vLLM 的 KV Cache 初始化流程（创建 tensor 后立即 `set_()` 替换底层存储）
- `aclrtMalloc` vs `aclrtMallocPhysical`：`aclrtMalloc` 分配的普通页（`ACL_MEM_MALLOC_NORMAL_ONLY`）需 ≥4KB 对齐，大页需 ≥2MB 对齐才支持 IPC。KV Cache block（~250KB）可通过 `HUGE_FIRST` 策略获得概率性 2MB 对齐
- 需要 Rust 侧新增 FFI 封装 `aclrtMalloc` / `aclrtFree`（`ascend.rs` 已有 `malloc_device` 函数，可直接复用）

#### 方案 C: 跟踪 vLLM Ascend 批量化 DMA 进展（低改动，等待上游）

**原理**：vllm-ascend PR [#7819](https://github.com/vllm-project/vllm-ascend/pull/7819) 正在实现 `aclrtMemcpyBatchAsync`（CANN ≥ 8.5.0）用于批量 KV Cache swap。该 PR 直接操作 vLLM 的 block 指针进行批量 D2H/H2D，绕过了 PegaFlow 的 DMA 需求。

**状态**：
- 该 PR 已合并，支持 CANN ≥ 8.5.0
- `aclrtMemcpyBatchAsync` 是 `aclrtMemcpyAsync` 的批量化版本，单次调用将多个 block 的拷贝合并为一个批量操作
- 性能提升显著：TTFT ~11% 降低，TPOT ~18% 降低

**对 PegaFlow 的影响**：
- 如果 vLLM Ascend 内置了 KV Cache swap 能力（通过 `aclrtMemcpyBatchAsync`），PegaFlow 的部分价值主张（D2H/H2D 传输）可能被替代
- 但 PegaFlow 的跨节点协调、MetaServer 哈希注册/查询、LRU 缓存管理、SSD 卸载等功能仍为差异化价值
- PegaFlow 可以与 vLLM Ascend 的 batch swap 接口合作——PegaFlow 提供跨节点缓存发现和协调，vLLM 提供本地 H2D/D2H DMA

#### 方案 D: 改造钉内存路径绕过 CANN ACL 的 DMA 引擎（极端方案，不推荐）

**原理**：在 Rust 侧使用 `aclrtMallocHost` 分配钉内存后，通过 PCIe BAR 映射直接进行 MMIO 读写 NPU HBM（完全绕过 ACL API）。

**缺点**：需要了解 Ascend 的 PCIe BAR 布局和 HBM 物理地址映射（非公开文档），开发周期长、维护风险高，不推荐。

## 九、Raw Pointer 模式还原——恢复 CANN IPC 零拷贝

### 9.0 问题分析：Raw Pointer 与 CUDA IPC 的本质差异

PegaFlow 源仓库的跨进程内存共享模型依赖 CUDA IPC：

```
vLLM Worker                          pegaflow-server
───────────                          ───────────────
torch.Tensor (GPU memory)
  → storage._share_cuda_()           → pickle.loads(wrapper)
  → CudaIPCWrapper(ipc_handle)       → wrapper.to_tensor()
  → pickle.dumps(wrapper)                   ↓
  → gRPC register_context_batch()    → torch.UntypedStorage
                                     → ._new_shared_cuda(handle)
                                            ↓
                                     真实的 torch.Tensor
                                     (共享同一块 GPU 物理内存)
                                     → data_ptr() 返回有效的设备地址
                                     → untyped_storage().nbytes() 返回真实大小
                                     → registry.rs 持有真实 tensor（生命周期管理）
```

Ascend 迁移版被迫降级为 raw pointer 模式：

```
vLLM Worker                          pegaflow-server
───────────                          ───────────────
torch.Tensor (NPU memory)
  → NpuIPCWrapper(tensor)            → pickle.loads(wrapper)
    → self.key = b""                 → wrapper.to_tensor()
    → self._raw_ptr = ptr                  ↓
    → self._raw_size = nbytes        → _RawDeviceProxy (NOT 真实 tensor)

                                     不能 set_()
                                     不能进行任何 tensor 操作

```

两者之间的四个根本差异：

| | CUDA IPC (source) | Raw Pointer (target) |
|---|---|---|
| **内存共享** | IPC handle 导出 → OS 级映射 → 真实共享 | 无共享——仅传一个地址数字 |
| **生命周期** | `Py<PyAny>` 持有 tensor → GC 管理 → 安全 | 无持有——raw ptr 可能随时变悬垂 |
| **to_tensor() 返回值** | 真实 `torch.Tensor`，可 `set_()`、可被 registry 持有 | `_RawDeviceProxy`（stub），仅元数据 |
| **跨节点能力** | ✅ IPC handle → dma-buf fd → RDMA | ❌ raw addr 仅在本地进程空间有效 |
| **UUID 设备识别** | ✅ GPU UUID 映射 → CUDA_VISIBLE_DEVICES 安全 | ❌ 硬编码 device_index（Ascend UUID 不可靠） |

具体侵入点：

1. **`NpuIPCWrapper.__init__`（第 206-226 行）**：强制走 raw pointer——`self.key = b""`，跳过了 CANN IPC key 的导出。CANN IPC 的 ctypes 调用链（`_npu_ipc_export_key_ctypes` → `aclrtIpcMemGetExportKey`）完整存在但**永远不会被执行**。

2. **`NpuIPCWrapper.to_tensor()`（第 228-282 行）**：`raw_ptr != 0` 分支总是 true（因为 `__init__` 总是设置 `_raw_ptr`），返回 `_RawDeviceProxy`。CANN IPC 导入分支（`self.key` → `aclrtIpcMemImportByKey` → 真实 tensor 重建）**永远不会执行**。

3. **`_RawDeviceProxy`（第 371-418 行）**：三个 stub 类（`_RawDeviceProxy`、`_DeviceStub`、`_UntypedStorageStub`）仅提供 `data_ptr()`、`device.index`、`nbytes()` 三个方法给 `registry.rs` 的 `materialize_tensor()` 读取——这些类**完全不属于**源仓库的设计。它们骗过了 registry 的类型检查，但 `registry.rs` 持有的 `Py<PyAny>` 不再是一个真正的 tensor。

4. **`device_index` hack**（`worker.py:316`）：`wrapper.device_index = self._ctx.device_id`——直接在序列化前篡改 wrapper 的 device_index 字段以匹配 context 的全局 device_id。这是绕过 Ascend NPU UUID 不可靠问题（全零 UUID）的 workaround。源仓库通过 GPU UUID 自动发现设备映射，不需要此类 hack。

5. **`_resolve_ipc_wrapper_factory`（`worker.py:1054-1063`）**：新增的设备类型分发函数，源仓库没有——源仓库直接 `CudaIPCWrapper(kv_cache)`。

6. **Tuple KV-cache flatten**（`worker.py:282-289`）：将 `(k_cache, v_cache)` tuple 展平为 `{layer_k, layer_v}` 两个独立键。这是 Ascend prefill-disaggregation 模式的适配（Ascend 返回 tuple，CUDA 返回单个 tensor），与 IPC 机制无关，但和 raw pointer 一起引入，增加了 worker.py 的复杂度。

### 9.7 还原后的跨进程内存共享架构

```

  CudaIPCWrapper(tensor)
    → storage._share_cuda_()  → CUDA IPC handle
    → pickle → gRPC → registry.rs
```

> **报告结论**: PegaFlow 的 Ascend 迁移在架构设计层面是高质量的工作——设备抽象层的 enum dispatch 设计干净，feature-gating 完整。但受限于 CANN `aclrtMemcpyAsync` 和 `aclrtIpcMemImportByKey` 要求 Device 侧内存来源于 `aclrtMallocPhysical`，而 PyTorch 缓存分配器底层并非通过此 API 分配（这与 `expandable_segments` 的开关状态无关），迁移中引入了两处根本性的架构偏离：
>
> 1. **Plan 3（Python 侧 D2H/H2D）**：核心数据传输链路从 Rust 侧零拷贝 DMA 退化为 Python 侧 `tensor.to('cpu')` / `tensor.copy_()`，绕过了整个 `TransferBackend` 抽象层。还原方案见 §8（覆盖 8 个文件，含 proto、engine core、server RPC、PyO3 bindings、Python connector）。
>
> 2. **Raw Pointer 模式（假 IPC）**：跨进程内存共享从 CANN IPC 零拷贝退化为直接传递设备虚拟地址——无共享、无生命周期管理、`_RawDeviceProxy` 伪造 tensor 接口。还原方案见 §9（覆盖 5 个文件，含 NpuIPCWrapper 重写、worker.py 分发移除、C 扩展接入、registry 命名还原）。
>
> 两处偏离的根本原因相同——PyTorch 缓存分配器 vs `aclrtMallocPhysical` 的内存来源不匹配。恢复 Rust 侧原生 DMA 和 CANN IPC 零拷贝有两条可行路径：短期推荐方案 B（`aclrtMalloc` 混合分配，仅对 KV Cache 层替换底层存储），中长期推荐方案 A（`camem_allocator` 全局物理内存分配，一次性恢复 DMA + IPC + RDMA + SSD 全部路径）。最关键的前置条件是将 KV Cache 的底层设备内存来源切换为 CANN DMA 兼容的分配路径。
